Importamos dos librerías: requests para hacer peticiones HTTP (hablar con la API), y psycopg2 para hablar con Postgres. Son librerías completamente independientes — una no sabe que existe la otra, tú eres quien conecta los datos de una a la otra.
pythonAPI_KEY = "660e85a76dfda5c0ca1cc05896bb4c4c"
url = "https://api.themoviedb.org/3/genre/movie/list"
params = {"api_key": API_KEY, "language": "es-ES"}
params es un diccionario con los parámetros que van en la URL. requests los añade automáticamente como query string, así que esto es equivalente a escribir a mano:
https://api.themoviedb.org/3/genre/movie/list?api_key=660e...&language=es-ES
Es más limpio y evita errores de codificar caracteres especiales tú mismo.
pythonresponse = requests.get(url, params=params)
data = response.json()
genres = data["genres"]
requests.get() hace la llamada HTTP y espera (bloquea el script) hasta que la API responde. response es un objeto con toda la info de la respuesta (código de estado, cabeceras, contenido...). .json() coge el cuerpo de la respuesta (que es texto plano) y lo convierte en una estructura de Python (dict/listas anidadas) — es lo mismo que hace json.loads(), pero como método directo sobre la respuesta.
data es un diccionario. Mirando el JSON que viste, tiene la forma {"genres": [{"id":28,"name":"Acción"}, ...]}, así que data["genres"] te da directamente la lista de diccionarios que nos interesa.

psycopg2.connect() abre una conexión TCP real a tu servidor Postgres (como hacía psql, pero desde código). El try/except es manejo de errores: si la conexión falla (contraseña mal, BD no existe, servidor caído...), en vez de que el programa se rompa con un traceback feo, capturamos la excepción (e) y decidimos qué hacer — en este caso, solo imprimirla. Un cursor es el objeto que realmente ejecuta comandos SQL sobre la conexión. Piensa en conn como "la línea telefónica abierta con Postgres" y cur como "quien habla por esa línea".

Recorremos la lista de géneros uno por uno. Por cada uno, ejecutamos un INSERT. Fíjate en la sintaxis: el SQL tiene %s como placeholders (marcadores de posición), y le pasamos una tupla (genre["id"], genre["name"]) como segundo argumento. Nunca metas los valores directamente dentro del string SQL con f-strings — psycopg2 se encarga de "escapar" los valores de forma segura para evitar inyección SQL (alguien metiendo código SQL malicioso a través de un dato).
ON CONFLICT (id) DO NOTHING significa: si ya existe una fila con esa id (recuerda que id es PRIMARY KEY), no hagas nada, sáltala sin dar error. Así puedes ejecutar el script mil veces sin que se rompa ni duplique datos.

Esto es crítico y muy fácil de olvidar. Postgres trabaja con transacciones: hasta que no haces commit(), los cambios están "en borrador", no persisten de verdad en la base de datos. Si se te olvida el commit(), puedes ejecutar el script, ver que no da error, y aún así no encontrar nada en la tabla al mirarla en pgAdmin.

    results = []
    for page in range(1, pages + 1):

            f"{BASE_URL}/movie/popular",

Aquí la diferencia principal con el script de géneros es que encapsulamos la lógica en una función en vez de tenerlo todo suelto. Esto es más profesional porque:

Puedes reutilizarla (llamarla varias veces con distinto número de páginas)
Puedes importarla desde otro archivo (from cargar_peliculas import get_popular_movies)
Es más fácil de testear de forma aislada

Dentro:

results = [] — creamos una lista vacía donde iremos acumulando todas las películas de todas las páginas
for page in range(1, pages + 1) — la API de TMDB pagina los resultados (20 películas por página), así que si quieres 100 películas necesitas pedir 5 páginas. range(1, pages+1) genera 1, 2, 3... hasta pages (si pusieras solo range(pages) empezaría en 0, y la API espera que la página empiece en 1)
response.raise_for_status() — si la petición HTTP falló (código 4xx o 5xx), esto lanza una excepción inmediatamente, parando el script con un error claro en vez de seguir con datos corruptos
results.extend(response.json()["results"]) — .extend() (a diferencia de .append()) añade cada elemento de la lista nueva a results, no la lista entera como un único elemento. Si response.json()["results"] tiene 20 películas, results crece en 20 elementos, no en 1

pythondef build_movie_genres(df):
    filas = []
    for _, row in df.iterrows():
        for genre_id in row["genre_ids"]:
            filas.append({"movie_id": row["id"], "genre_id": genre_id})
    return pd.DataFrame(filas)
Esta es la parte más conceptualmente interesante. El problema: la columna genre_ids de cada película contiene una lista (ej: [27, 53]), pero tu tabla movie_genres necesita una fila por cada combinación película-género (una fila con movie_id=X, genre_id=27 y otra fila movie_id=X, genre_id=53). Esto se llama "explotar" una columna de listas en filas.

for _, row in df.iterrows(): recorre el DataFrame fila por fila. iterrows() devuelve pares (índice, fila) — usamos _ para el índice porque no nos interesa, solo row (los datos de esa fila)
for genre_id in row["genre_ids"]: por cada película, recorremos su lista de géneros (bucle anidado dentro del bucle de filas)
filas.append({"movie_id": row["id"], "genre_id": genre_id}): por cada combinación, añadimos un diccionario nuevo a la lista filas
Al final, pd.DataFrame(filas) convierte esa lista de diccionarios en un DataFrame de dos columnas (movie_id, genre_id), con tantas filas como combinaciones película-género existan

Ejemplo mental: si la película 1339713 tiene genre_ids: [27, 53], esto genera dos filas: {movie_id: 1339713, genre_id: 27} y {movie_id: 1339713, genre_id: 53}.

Aquí pasamos de "DataFrame de pandas" a "lista de listas de Python plano", que es lo que execute_values espera. df[columnas_bd] selecciona esas columnas en ese orden exacto (importante: tiene que coincidir con el orden de columnas del INSERT). .values te da un array de numpy, y .tolist() lo convierte a una lista normal de Python, tipo:

python    execute_values(cur, """
        INSERT INTO movies (...)
        VALUES %s
        ON CONFLICT (id) DO UPDATE SET
            popularity = EXCLUDED.popularity,
            vote_average = EXCLUDED.vote_average,
            vote_count = EXCLUDED.vote_count
    """, registros)
execute_values es una utilidad especial de psycopg2 (por eso la importamos aparte con from psycopg2.extras import execute_values) que construye automáticamente un único INSERT con todos los registros a la vez, en vez de hacer un INSERT por fila en un bucle. Es mucho más rápido cuando insertas muchas filas.
ON CONFLICT (id) DO UPDATE SET ...: como ya vimos, si la película ya existe (mismo id), en vez de fallar o ignorarla, actualiza esos 3 campos con los valores nuevos. EXCLUDED es una palabra especial de SQL que significa "el valor que se intentó insertar" — o sea, EXCLUDED.popularity es la popularidad nueva que traías en registros, y la usamos para pisar la que ya había en la tabla.
Por qué solo actualizamos esos 3 campos y no todo: porque popularity, vote_average y vote_count son datos que cambian con el tiempo (una película se vuelve más o menos popular), mientras que title, overview, etc. no cambian, así que no tiene sentido reescribirlos cada vez.
